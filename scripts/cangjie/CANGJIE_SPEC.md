# 仓颉方案技术规范

本文档记录 Sicang5/Wucang5 的生成逻辑、简码分层策略与当前默认参数。

## 1. 任务背景

仓颉系列方案的目标是在保留仓颉五代取码直觉的前提下，用现代语料做高频优先排序与简码分配。Wucang5 是五码单字流方案；Sicang5 是四码单字流方案。

## 2. 核心数据源

- **Dialogue 口语语料**：`schemas/common/frequency/char/sc/dialogue_char_freq.txt`，权重 6
- **Subtlex 字幕语料**：`schemas/common/frequency/char/sc/subtlex_char_freq.txt`，权重 5
- **知乎语料**：`schemas/common/frequency/char/sc/zhihu_char_freq.txt`，权重 4
- **北语大语料**：`schemas/common/frequency/char/sc/blcu_char_freq.txt`，权重 2
- **Essay 语料**：`schemas/common/essay-zh-hans.txt`，权重 1

所有构建脚本通过 `cangjie_builder.get_weighted_frequencies()` 共享同一套加权字频，保证简码选择与最终字典排序使用同一份综合分。

## 3. 核心脚本

| 脚本 | 职能 |
| :--- | :--- |
| `core/gen_shortcut_1.py` | 一简校准生成与报告输出 |
| `core/gen_shortcut_2.py` | 二简净收益生成，支持固定数量、覆盖率、GB2312 保护 |
| `core/gen_shortcut_3.py` | 三简净收益生成，支持固定数量、覆盖率、GB2312 保护 |
| `core/gen_shortcut_4.py` | 四简生成，支持 `safe` / `balanced` / `aggressive` |
| `core/cangjie_builder.py` | 最终字典构建兼容入口 |
| `core/dict_builder.py` | 最终字典构建与排序 |
| `core/dedup.py` | `z` 后缀与自然 `z/x` 前缀消重 |
| `gen_wucang5.py` | Wucang5 一键构建入口 |
| `gen_sicang5.py` | Sicang5 一键构建入口 |

## 4. 简码原型文件

- 字根/特殊键本体码：`scripts/cangjie/prototypes/root_code.txt`
- 一简：`scripts/cangjie/prototypes/one_code.txt`
- 固定前缀简码：`scripts/cangjie/prototypes/fixed_prefix_code.txt`
- 二简：`scripts/cangjie/prototypes/two_code.txt`
- 三简：`scripts/cangjie/prototypes/three_code.txt`
- 前缀避重三码：`scripts/cangjie/prototypes/dedup_prefix_code_3.txt`
- 四简：`scripts/cangjie/prototypes/four_code.txt`（启用 `--s4` 时生成）
- 前缀避重四码：`scripts/cangjie/prototypes/dedup_prefix_code_4.txt`

旧的 `z_code.txt` 只作为过渡文件名保留，不再承载新的 `z?`、`z??` 或 `z???` 规则；现有 `zz:重` 这类特殊键本体码应迁入 `root_code.txt`。

## 5. Wucang5 默认构建

默认命令：

```powershell
python scripts/cangjie/gen_wucang5.py
```

当前默认参数：

- `--s2-count 300`
- `--s3-count 800`
- `--s2-coverage 0`
- `--s3-coverage 0`
- `--protect-native-min-score 3000`
- `--shortcut-candidate-min-score 3000`
- `--no-s4`（默认关闭；可用 `--s4` 开启）
- `--s4-mode balanced`
- `--s4-count 1000`
- `--s4-level2-min-score 1000`

含义：

- 二简、三简默认按固定数量生成；只有当 `--s2-count 0` 或 `--s3-count 0` 时，对应覆盖率参数才参与控量。
- 二简默认取全码前两码，三简默认取全码前三码。
- 生产构建默认 `--weights sc_daily`，用于二简、三简、全码排序、`z/x` 后缀救援，并在显式开启 `--s4` 时用于四简。
- 一简不由生产构建重算；`core/gen_shortcut_1.py` 的一简审阅报告默认使用 `--weights sc`。
- 二简、三简的候选池按 Rime 默认 `extended_charset=常用` 状态过滤；Ext-A、Ext-B 及以上和兼容汉字等增广字符不会参与简码分配。
- `--protect-native-min-score` 控制原生二三码位保护门槛。
- `--shortcut-candidate-min-score` 控制二三简长码候选的最低入选综合字频。
- 四简默认关闭；需要实验四简时显式传入 `--s4`。
- 四简启用时只处理 GB2312 一级字和达到频率门槛的 GB2312 二级字，不把全 CJK 字集压入四码层。
- 四简启用时默认上限 1000 个；`--s4-count 0` 表示不做数量截断，由四简模式自身过滤规则决定数量。
- GB2312 二级字启用四简时默认需要综合字频达到 1000；`--s4-level2-min-score 0` 表示不过滤二级字。

常用命令：

```powershell
# 开启四简
python scripts/cangjie/gen_wucang5.py --s4

# 四简安全版
python scripts/cangjie/gen_wucang5.py --s4 --s4-mode safe

# 四简平衡版，启用四简时默认
python scripts/cangjie/gen_wucang5.py --s4 --s4-mode balanced

# 四简激进版
python scripts/cangjie/gen_wucang5.py --s4 --s4-mode aggressive

# 固定四简数量
python scripts/cangjie/gen_wucang5.py --s4 --s4-count 1000

# 调整 GB2312 二级字进入四简的门槛
python scripts/cangjie/gen_wucang5.py --s4 --s4-level2-min-score 1000

# 覆盖率控量
python scripts/cangjie/gen_wucang5.py --s2-count 0 --s2-coverage 0.85 --s3-count 0 --s3-coverage 0.90
```

## 6. 简码分层策略

### 6.1 一简与 z 补丁

一简由 `core/gen_shortcut_1.py` 单独生成校准表，生产构建脚本只消费 `one_code.txt`，不自动重算一简。一简审阅默认使用 `--weights sc`，按现代简体日用频率和记忆锚点选候选。

```text
判断依据 = 频率倍数 + 省码收益 + 字根位代价
```

当前校准在原高频一简表上做小范围调整：`l: 个 -> 中`、`p: 也 -> 心`。`g: 去` 与 `r: 可` 经无人工定稿偏置的 `--blind` 审计后仍为日常频率与记忆锚点下的最优项。若字根字未被一简校准表选中，最终字典会过滤该字的原生一简入口，让它通过 `az/azz/bz...` 这类 z 字根码输入；若字根字在校准表中，则允许保留为一简。

### 6.2 二简

二简候选来自全码长度大于 2 的字：

- 默认取码：`full_code[:2]`
- 可选取码：首尾码
- 排序依据：净收益 `综合字频 × (原码长 - 2) - 原生码位代价`
- 权重：随生产构建 `--weights`，默认 `sc_daily`
- 默认模式：保护高频 GB2312 原生二码位，长码字可与其他原生二码位竞争
- 原生码位保护门槛：`--protect-native-min-score`，默认 `3000`
- 长码候选门槛：`--shortcut-candidate-min-score`，默认 `3000`
- 原生二码字是 GB2312 且达到门槛时受保护；长码字达到同一门槛才可入选二简
- `--no-protect-native`：允许长码字与原生二码字竞争，原主频次按 `1.5x` 计入净收益代价
- `--gb-only` 模式：仅 GB2312 字有资格，且 GB2312 原生二码位绝对保护

### 6.3 三简

三简候选来自全码长度大于 3、且未获得更短简码的字：

- 默认取码：`full_code[:3]`
- 可选取码：前两码 + 末码
- 排序依据：净收益 `综合字频 × (原码长 - 3) - 原生码位代价`
- 权重：随生产构建 `--weights`，默认 `sc_daily`
- 默认模式：保护高频 GB2312 原生三码位，长码字可与其他原生三码位竞争
- 原生码位保护门槛：`--protect-native-min-score`，默认 `3000`
- 长码候选门槛：`--shortcut-candidate-min-score`，默认 `3000`
- 原生三码字是 GB2312 且达到门槛时受保护；长码字达到同一门槛才可入选三简
- `--no-protect-native`：允许长码字与原生三码字竞争，原主频次按 `1.2x` 计入净收益代价
- `--gb-only` 模式：仅 GB2312 字有资格，且 GB2312 原生三码位绝对保护

### 6.4 四简

四简候选来自 GB2312、五码、且未获得 z/一简/二简/三简的字：

- 取码：`full_code[:4]`
- 权重：随生产构建 `--weights`，默认 `sc_daily`
- 启用后默认模式：`balanced`
- 默认关闭；开启方式：`--s4`
- 字集门槛：GB2312 一级字直接参与；GB2312 二级字需达到 `--s4-level2-min-score`

四简模式：

- `safe`：不压原生四码位，只接受没有原生四码冲突的候选。
- `balanced`：默认模式。若冲突原主已经有更短简码则放行；否则候选综合频次需超过活跃原主 `3.0x`。
- `aggressive`：GB 五码字全量截四码，保留内部重复，覆盖最大但候选压力也最大。

在 `safe` 与 `balanced` 下，四简会做码位去重与同字去重：每个四码位只保留一个最高价值字，同一个字也只保留一个四简。

## 7. 字典排序

最终字典使用纯 `字\t码` 格式，不写 weight。候选优先级依赖文件内排序：

1. 简码条目：`z` / 一简 / 二简 / 三简 / 四简（启用时）
2. 全码条目：同码内按“常用字优先、无普通简码字优先、同组内综合字频降序”排序
3. 消重条目：自动追加的 `z/x` 后缀码、自动 `z/x` 前缀短码和四码前缀选重码

这个排序保证普通简码在自身码位上拥有绝对首选；全码位参考自然码的简码让位，但只让给同码内仍依赖全码输入的常用字。`z/x` 后缀救援、自动前缀救援和 `fixed_prefix_code.txt` 固定前缀层都不触发全码让位。

## 8. 消重直达码

`core/dedup.py` 会在普通简码和全码让位之后，为重码候选字（跳过首选和已拥有普通简码的字）自动生成消重直达码。生成顺序为：短码 `z/x` 后缀 -> 自动 `z/x` 前缀短码 -> 四码 `z/x` 前缀选重码。

### 8.1 默认模式（短码 z/x 后缀）

后缀救援只处理短于方案最大码长的当前码位，默认跳过已有普通简码的字：

- Sicang5 默认只处理源码长度 `<= 3`：第 2 候选生成 `原码 + z`，第 3 候选生成 `原码 + x`；四码全码暂不自动救援。
- Wucang5 默认与 Sicang5 使用同一套后缀规则，但源码长度上限为 `4`：第 2 候选生成 `原码 + z`，第 3 候选生成 `原码 + x`。
- 候选字集通过 `--suffix-z-charset` 控制，默认使用 `frequency`。
- 最低综合字频通过 `--suffix-z-min-score` 控制。
- 候选位与后缀映射通过 `--suffix-z-ranks` 控制，例如 `2:z,3:x`。
- 源码最大长度通过 `--suffix-z-max-source-length` 控制。

默认占用策略为 `--suffix-z-occupied-policy ignore-nonfrequency-or-shortcut`：目标码若只对应 frequency 范围外的字，或对应字已经有普通简码，则视为可占用；否则跳过。

### 8.2 自动 z/x 前缀

启用 `--dedup-prefix` 时，前缀层自动生成，不再依赖人工定稿。该层分两步：

1. **前缀短码层**：为高频、无普通简码、当前非首选的字分配可由原码推导的 `z/x` 前缀码。
   - 二键层：主体键按普通一简锚点规则选择，依次考虑字根、首码、尾码和高频包含码，再尝试 `z + 主体键`、`x + 主体键`；主体键覆盖 `a-z`，已被字根/普通简码占用的 `xz/zz` 等仍自动跳过；默认用 `--dedup-prefix-level2-weights sc`，按全局高频重码痛点竞争，不要求节省码长。
   - 三键层：依次尝试 `z + 前两码`、`x + 前两码`；对标普通二码，默认用 `--dedup-prefix-level3-weights sc_daily`，按高频痛点并适度奖励节省码长。
   - 二键层和三键层使用四码投影基线生成，Sicang5 与 Wucang5 共享同一套审阅文件。
2. **方案专属深层前缀层**：排除已获得共享短前缀的字，再按方案最大码长生成：
   - Sicang5 四码前缀选重层只处理四码全码第 2/3 候选：
     - 第 2 候选：`z + 前三码`
     - 第 3 候选：`x + 前三码`
   - Wucang5 四码前缀短码层按 `z/x + 前三码` 生成，规则类似三键前缀短码，不复用 Sicang5 的四码选重结果。
   - Wucang5 五码前缀选重层才对应 Sicang5 四码选重层，只处理五码全码第 2/3 候选：第 2 候选 `z + 前四码`，第 3 候选 `x + 前四码`。

前缀层不扫描任意空码位：二码前缀使用普通一简锚点主体键，三码、四码和五码前缀使用与普通简码投影一致的前两码、前三码、前四码。目标码若被字根、普通简码、后缀救援或更早前缀层占用，则跳过；若只被 frequency 范围外字或已有普通简码字的全码占用，则可覆盖。

候选字集通过 `--dedup-prefix-charset` 控制：

- `frequency`：默认值，仅处理当前综合字频中出现的字。
- `gbk`：处理 GBK 可编码字符。
- `gb2312`：仅处理 GB2312 汉字。
- `all`：不限制汉字范围。

最低综合字频通过 `--dedup-prefix-min-score` 控制，默认值为 `1`。前缀短码层级通过 `--dedup-prefix-short-levels` 控制；是否启用短码层和深层选重层分别由 `--dedup-prefix-short`、`--dedup-prefix-full` 控制。前缀层权重独立于主构建 `--weights`，默认分层为 `z?/x?=sc`，`z??/x??` 及更深层为 `sc_daily`。

### 8.3 结构规则模式（Structure Rule-based Suffix）

通过 `--suffix-structure` 参数启用。本模式使用汉字的**间架结构**（由 IDS 字根描述序列首字符决定）作为消重后缀，完全不依赖读音，适合不知读音也能盲打识形的仓颉用户。

> **默认策略**：Sicang5 默认启用短码 `z/x` 后缀和自动 `z/x` 前缀；不加载 `fixed_prefix_code.txt`，不启用结构后缀。结构后缀保留为可显式开启的实验模式。

#### 结构数据来源

结构信息读取自 `scripts/ids/ids.txt`，该文件每行格式为：

```
U+xxxx <字符> <IDS描述>
```

取每个字 IDS 描述的**首字符**（IDS 运算符）来判断结构类别。

#### 结构分类与键位映射

| 结构类别 | IDS 运算符 | 后缀键 | 手位 | 记忆关联 |
| :--- | :--- | :---: | :---: | :--- |
| 左右 / 左中右 | `⿰` `⿲` | `z` | 左 | 左右结构最常见，使用最低频保留键 |
| 上下 / 上中下 | `⿱` `⿳` | `x` | 左 | 上下结构次常见，使用低频下排键 |
| 包围 / 半包围 | `⿴` `⿵` `⿶` `⿷` `⿸` `⿹` `⿺` `⿻` | `w` | 左 | 围合结构取 wrap 的首字母联想 |
| 独体 / 单一字根 | 其他（非 IDS 运算符开头） | `a` | 左 | 单体归入基础键 |

> **说明**：结构后缀默认键为 **`z x w a`（即 "zxwa"）**，可用 `--suffix-structure-keymap` 调整。四个键不要求连续；当前默认来自 Sicang5 全码表键频统计，使用 `z/x/w/a` 这组低频键，并优先照顾结构记忆性。所有四个结构后缀键均位于**左手**，配合四码仓颉本身右手偏重（约 58%）的键位分布，改善整体手指负荷平衡。

#### 生成规则

1. **免加后缀条件**（同拼音模式）：
   - 候选列表中第 1 候选（首选）不加后缀。
   - 已拥有简码的字不加后缀。
   - 当前重码编码长度已达到或超过方案最大码长时不加后缀。
2. **候选字集过滤**（结构模式专有）：
   - 默认使用 **GBK**：保留常见繁简汉字，避免把大量 Extension B/C/D/E 等罕用字压进结构后缀层。
   - 可选 `all`：不限字集，覆盖最大，但会明显伤害繁体/大字符集动态选重率。
   - 可选 `gb2312`：极致简体优化，简体选重率最低，但基本不考虑繁体体验。
3. **目标码位保护**（结构模式专有）：
   - 默认使用 `protect-min-score`：若拟生成的后缀码已是简码码位，或既有码位首选字综合字频达到 `100000`，则放弃生成；低频全码占用可被结构后缀压过。
   - 可选 `skip-any`：只要拟生成后缀码已被任何条目占用就放弃，最保守，但消重收益较低。
   - 结构后缀自身仍保持同一码位只生成一个字，不让新增结构码互相重码。
4. **兜底**：若 `ids.txt` 中无此字的结构信息，则不生成后缀。

#### 典型示例

| 字 | 原码 | 结构 | 后缀键 | 结构直达码 |
| :---: | :---: | :---: | :---: | :---: |
| 杠 | `dm` | 左右 | `a` | `dma` |
| 杢 | `dm` | 上下 | `s` | `dms` |
| 扌 | `dm` | 独体 | `f` | `dmf` |
| 井 | `tt` | 包围 | `d` | `ttd` |
| 卌 | `tt` | 独体 | `f` | `ttf` |
| 匕 | `um` | 独体 | `f` | `umf` |
| 屵 | `um` | 上下 | `s` | `ums` |
| 屸 | `um` | 左右 | `a` | `uma` |

经数据测算（针对 Sicang5，≤3 码全部冲突字，不限 GBK，共 5922 个），旧 `skip-any` 无损安全约束下可生成约 **3066 个** 独占无碰撞的结构直达码，覆盖约 **51.8%** 的目标冲突字。

结构实验推荐参数为 `GBK + protect-min-score=100000 + zxwa`。`GB2312 + protect-min-score` 可进一步压低简体，但繁简联合会上升，属于极致简体优化选项。`all + protect-min-score` 虽覆盖更多，但繁体和繁简联合明显恶化，不作为默认。

Sicang5 全码表键频统计（由 `cangjie5.dict.yaml` 投影到四码、去重、无简码/无后缀）：

| 字集 | 最低频键（从低到高） |
| :--- | :--- |
| 全汉字 | `z:21` `x:1249` `w:9112` `q:9460` `a:9468` `p:12399` `c:12568` `v:13081` |

## 9. 简码避重优化设计

Sicang5 与 Wucang5 的后续优化方向不再追求给 GBK 全范围汉字生成避重简码，而是直接服务 `summary.py` 的五个动态选重指标：

- 知乎简体：`schemas/common/frequency/char/sc/zhihu_char_freq.txt`
- 北语简体：`schemas/common/frequency/char/sc/blcu_char_freq.txt`
- 台标繁体：`schemas/common/frequency/char/tc/taiwan_char_freq.txt`
- 古籍繁体：`schemas/common/frequency/char/tc/guji_char_freq.txt`
- 繁简联合：以上简繁语料的联合效果

候选池只需要覆盖这些频表中出现的字。GBK/GB2312 仍可作为安全过滤条件或兼容性观察指标，但不再作为“必须全量出简避重”的目标。

固定特殊简码层默认采用 **简体 60% / 繁体 40%** 的 `sc60/tc40` 权重，而不是纯简体、纯繁体或 50/50。依据：

- OpenCC `t2s` 精确统计显示，台标繁体语料中传承字频率约 **67.90%**，古籍繁体约 **73.07%**；繁体环境中有大量输入频率落在简繁同形字上，简体高频权重不会完全浪费。
- `fixed-special-shortcut` 临时实验中，`sc60/tc40` 在 Wucang5 上与 `sc65/sc70/sc75` 简体表现相同，并取得最低繁简联合选重率；在 Sicang5 上，`sc70` 虽进一步压低简体选重率，但台标、古籍、繁简联合均比 `sc60` 略差。
- 因此固定特殊简码层选择 `sc60/tc40` 作为通用默认；若未来需要极致简体体验，可另开简体专精分支，而不覆盖通用默认。

### 9.1 基本原则

1. **保留现有 z 后缀规则不动**  
   当前 `原码 + z` 只处理尚未达到方案最大码长的重码组第二候选，规则简单、记忆成本低、收益稳定。该层继续作为低码长第二候选的直达补丁。

2. **用高频简码优先消除真实语料重码**  
   简码分配目标从“覆盖字符集”转为“压低动态选重率”。优先给四个评估频表中的高频字生成简码，让真实输入中最常遇到的重码先消失。

3. **出了简码的字，全码让位**  
   一个字获得简码后，它在原全码位置不应继续占据首选。该字的全码条目应降到满足 `is_common_han_char` 过滤条件的候选末位。这样全码首选位置释放给没有简码、仍依赖全码输入的字。

4. **控制全码候选压力**  
   目标是在排除“已有简码字”的干扰后，尽可能让全码候选数控制在 2 个以内：
   - 未达到方案最大码长的重码组：第二候选可通过 `原码 + z` 直达。
   - 已达到方案最大码长的重码组：第二候选可用分号 `;` 选择，手感优于第三候选的 `'`。
   - 因此策略重点是避免高频字落到第三候选及以后，尤其是方案最大码长的全码位。

5. **后缀优先，前缀分层**
   Sicang5 先使用短于四码的 `z/x` 后缀救援：第 2 候选补 `z`，第 3 候选补 `x`。随后自动生成 `z?/x?`、`z??/x??` 高频前缀短码，最后再用 `z???/x???` 处理四码全码第 2/3 候选。前缀短码层优先于四码前缀选重层，四码层必须排除已获更短前缀码的字。

### 9.2 固定一简与 z/x 救援分层

固定高价值简码不再把一简、繁体专精、重码救援混在同一层处理：

1. **`a-z` 主一简层**
   - 承担最高频、最高收益的一简输入。
   - 人工定稿后保持稳定；生成脚本只提供候选报告，不自动覆盖定稿表。
   - 简体/通用方案消费 `one_code.txt`；繁体专精方案应消费独立的 `one_code_tc.txt`，而不是用 `z?` 承载繁体高频一简。

2. **`z/x` 后缀重码救援层**
   - 在普通二三简和全码让位之后生成，不参与普通简码竞争。
   - Sicang5 默认只救短于四码的当前码位：第 2 候选补 `z`，第 3 候选补 `x`。
   - 目标码若只被 frequency 范围外字占用，或占用字已有普通简码，可视为可占用。
   - 该层不触发全码让位。

3. **自动 `z/x` 前缀层**
   - `z?/x?` 和 `z??/x??` 用作高频前缀短码，目标是同时降重和降码长。
   - `z???/x???` 用作四码前缀选重码，目标是把四码全码第 2/3 候选变为直达。
   - 该层全自动生成，不作为繁体高频补位层，也不作为另一套一简层。
   - `fixed_prefix_code.txt` 仍可通过 `--fixed-prefix` 显式加载，但不属于默认构建。

#### 简繁一简分表

简体/通用一简和繁体专精一简分别人工定稿，不通过 `z?` 做主副层对调：

```text
one_code.txt      简体/通用一简
one_code_tc.txt   繁体专精一简
```

构建简体方案时读取 `one_code.txt`；构建繁体专精方案时读取 `one_code_tc.txt`。繁体日常高频字如 `的/不/我/是/有/了` 应进入繁体一简主层，而不是占用 `z?` 固定前缀码。

普通二简、三简、四简仍按方案切换语料权重。`z/x` 后缀使用主构建综合分排序；自动前缀使用独立分层权重：`--dedup-prefix-level2-weights sc`、`--dedup-prefix-level3-weights sc_daily`、`--dedup-prefix-full-weights sc_daily`。frequency 范围只作为候选/冲突过滤字集边界，不改变各层指定的排序口径。

#### 原型文件分层与加载顺序

固定简码和避重补码按来源与稳定性分文件，不再用 `z_code.txt` 表示所有 z 开头编码：

| 文件 | 内容 | 维护方式 |
| :--- | :--- | :--- |
| `root_code.txt` | 字根字、特殊键本体码，例如 `重	zz` | 极少改动，人工维护 |
| `one_code.txt` | 单键一码直达，包含最终定稿的 `a-z` | 人工定稿 |
| `fixed_prefix_code.txt` | 旧固定 `z?`、`x?` 二码补丁；默认不加载 | 显式启用 |
| `prefix_code_2.txt` | `z?`、`x?` 高频前缀短码审阅产物 | 构建末尾自动覆盖生成 |
| `prefix_code_3.txt` | `z??`、`x??` 高频前缀短码审阅产物 | 构建末尾自动覆盖生成 |
| `prefix_code_4_sicang5.txt` | Sicang5 `z???`、`x???` 四码第 2/3 候选前缀选重码审阅产物 | 构建末尾自动覆盖生成 |
| `prefix_code_4_wucang5.txt` | Wucang5 `z???`、`x???` 四码前缀短码审阅产物 | 构建末尾自动覆盖生成 |
| `prefix_code_5_wucang5.txt` | Wucang5 `z????`、`x????` 五码第 2/3 候选前缀选重码审阅产物 | 构建末尾自动覆盖生成 |
| `suffix_code_sicang5.txt` | Sicang5 `z/x` 后缀救援码审阅产物 | 构建末尾自动覆盖生成 |
| `suffix_code_wucang5.txt` | Wucang5 `z/x` 后缀救援码审阅产物 | 构建末尾自动覆盖生成 |

自动前缀和后缀仍由构建算法决定，`prefix_code_*.txt` 与 `suffix_code_*.txt` 是审阅产物，不是人工输入源；下次构建会覆盖。`prefix_code_2.txt` 和 `prefix_code_3.txt` 默认使用 `--dedup-prefix-source-max-code-length 4` 的四码投影基线，保证 Sicang5/Wucang5 的二三码前缀一致；四码及以上前缀按方案独立生成。前缀权重默认对标普通简码层：二码前缀对标一简用 `sc`，三码和更深层前缀用 `sc_daily`。

推荐加载顺序：

```text
root_code.txt
one_code.txt
two_code.txt
three_code.txt
four_code.txt
z/x suffix rescue
z/x short prefix
z/x four-code prefix
```

生成后续层时必须排除前面所有已占用的码位和汉字。`z/x` 后缀救援在普通简码和全码让位之后生成；自动前缀在后缀之后生成。`fixed_prefix_code.txt` 若显式启用，也不触发全码让位。

### 9.3 简码挑选评分

候选池初筛仍以语料加权频率为主体，同时加入“全码是否非首选”的惩罚/奖励项：

```text
score = 语料加权频率收益 + 全码非首选奖励 + 候选位降深收益 - 码位占用代价
```

其中：

- **语料加权频率收益**：来自知乎、北语、台标、古籍四个评估频表；权重应以 `summary.py` 的目标指标为准，而不是只看综合字频。
- **全码非首选奖励**：如果该字当前全码不是首选，同等频率下应提高优先级，因为它真实输入时已经需要选重。
- **候选位降深收益**：如果简码能把一个高频字从第 3 候选及以后移出，收益高于只把第 2 候选移出。
- **码位占用代价**：若目标简码位已有高频首选、已有简码、或会制造新的高频重码，应扣分或放弃。

`core/gen_shortcut_1.py` 负责一简候选报告：先用静态评分缩小每键候选池，再调用 `core/shortcut_gain.py` 重放真实后续层。`core/shortcut_gain.py` 是可复用的净收益核心，也可单独用于人工复核：

```text
root_code
-> one_code
-> fixed_prefix_code
-> 按当前保护参数重建 S2
-> 排除 S2 后重建 S3
-> 回退到最短可用路径或全码
```

对一个固定简码新增或替换，最终净收益定义为：

```text
总净收益 = Σ 字频 × (变更前最短可用码长 - 变更后最短可用码长)
         = 直接收益 + S2/S3 联动收益
```

- **直接收益**：目标码位新旧两个字自身的码长变化。
- **S2/S3 联动收益**：新旧字退出或进入普通简码层后，其他字因码位释放、原生码位恢复、候选竞争变化产生的净收益。
- **保护参数必须与目标方案一致**：例如默认 `--protect-native-charset gbk` 下，`戶:hs` 会保护 `hs`，因此 `所` 的真实普通简码回退是 `hsh`，不能按 `hs` 估算。
- **固定前缀复用**：同一分析器可评估 `z?`、`x?` 二码空位新增或既有码位替换；只需把目标层改为 `fixed-prefix`。

常用命令：

```powershell
# 一简替换：评估 t: 着 -> 其
python scripts/cangjie/core/shortcut_gain.py --layer one --code t --char 其 --weights sc

# 固定前缀二码：评估 xp 码位新增或替换为某字
python scripts/cangjie/core/shortcut_gain.py --layer fixed-prefix --code xp --char 恐 --weights sc_daily
```

权重通过 `--weights` 显式选择：

- `sc`：现代简体日用优化，使用口语 20.75%、字幕 27.57%、知乎 27.92%、北语 23.76%；比例由 `core/optimize_sc_weights.py` 最小化现代简体语料共识的平均 Jensen-Shannon 距离得到。`gen_shortcut_1.py` 与 `shortcut_gain.py --layer one` 默认使用该模式。
- `sc_daily`：简繁日常通用，使用知乎 44%、北语 36%、台标 11%、古籍 9%；普通二三简、自动前缀和固定前缀评估默认使用该模式。
- `sc_balanced`：简繁平衡，使用知乎 33%、北语 27%、台标 22%、古籍 18%；可用于偏繁体/古籍均衡的实验构建。

一简不直接套用普通二三简的简繁日常通用排序。一简位数量少、日常曝光高，候选报告应以 `sc` 日常频率作为默认初筛与重放口径。`a-z` 一简的静态评分只使用日常字频和记忆锚点，不加入省键数、长码压力、候选深度或前缀占位等重码救援权重；再由一简决策层按 `字根 > 首码 > 尾码 > 包含码` 的锚点等级筛选，同级内优先保留日常频率更高的字。`x/z` 没有普通仓颉锚点，使用 `--append-xz` 追加时按日常字频生成全局候选。跨键分配时，同一个字只能在最合理的键位占据一次 `Rank 1`，在其他键位只能作为 `Rank 2+` 备选。真实净收益用于展示副作用和排除负收益替换，不直接压过更强锚点或更高日常频率。普通二三四简和自动消重层仍保留重码救援评分；自动 `z/x` 前缀默认使用 `sc_daily`，60/40 的 `sc_balanced` 保留为显式实验口径。

需要执行无人工定稿偏置的逐键替换审计时，运行 `gen_shortcut_1.py --blind --output <path>`。盲测仍以当前正式版作为替换净收益基线和报告对照，但当前字不再获得静态候选保底；若它凭自身频率和锚点自然进入短名单，则以收益 `0` 的基线正常参与主推荐排序。该模式不是从空白状态独立生成完整一简方案。

同一 Unicode 字可能因地区字形存在多个普通仓颉码。所有简码生成器和 `shortcut_gain.py` 在分析一字多码时，都先按当前 `--weights` 模式名经过 `core/glyph_codes.py` 过滤：模式名包含 `sc` 时加载 `data/sc_glyph_preferred_code.txt`，否则模式名包含 `tc` 时加载 `data/tc_glyph_preferred_code.txt`。例如 `sc_balanced` 虽然混入繁体语料，仍按 `sc` 大陆字形处理。尚未建立的区域表按空表处理。最终字典仍保留兼容全码，但自动 `z` 后缀、自然 `z/x` 前缀和结构后缀只从过滤后的地区字形视图派生。

大陆字形首选码表 `data/sc_glyph_preferred_code.txt` 只处理 GBK 范围内的一字多码汉字，并排除最终结果中的 `x/z` 前缀编码：先从 `data/cj5-90000.txt` 与 Cangjie5 源码表的唯一普通码交集离线提取，再通过 `core/fetch_sc_glyph_preferred_codes.py` 分批查询教育大学汉语多功能字库中带 `GBK` 的字形记录补齐剩余项。在线结果优先采用唯一 GBK 普通码；若 GBK 码带 `x/xx...` 前缀，则先尝试去掉前缀后直接命中源码候选，再尝试将其唯一展开为以该缩略码开头的源码候选。仍无法唯一判断的字写入 `data/sc_glyph_unresolved_code.txt` 供人工确认，不应自动选择最短兼容码作为大陆字形退路。`--sort-by-code` 是独立的纯排序操作，只整理现有输出文件，不读取码表、合并缓存或访问网络。

该重放算法衡量的是码长层面的净收益，用于排除明显低效方案；最终定稿仍需结合键位联想、简繁展开方式和 `summary.py` 的五个动态选重指标。

### 9.4 全码排序规则

最终字典排序需要配合简码策略调整：

1. 简码条目仍然优先输出，保证简码码位首选。
2. 全码条目按“未获简码字优先、已获简码字让位”的原则排序。
3. 已获简码字不删除全码，但在 `is_common_han_char` 允许范围内降到该全码候选末位。
4. 非常用汉字不应因为让位规则被主动提前；全码首选应优先留给常用且无简码的字。

该策略的核心判断标准是 `scripts/assess/summary.py` 中五个动态选重率是否下降，而不是新增简码数量本身。

## 12. 词组取码动态配额

Sicang5 语句流采用动态配额算法，根据单字码长分配 4 码位。

| 词长 | 结构配额 | 取码明细 | 示例 |
| :--- | :--- | :--- | :--- |
| 二字词 | 2 + 2 | A首+A尾 + B首+B尾 | 实际 -> pmnf |
| 二字词 | 1 + 3 | A首 + B首+B次+B尾 | 中国 -> lwim |
| 二字词 | 2 + 1 | A首+A尾 + B首 | 某个 -> mwg |
| 三字词 | 2 + 1 + 1 | A首+A尾 + B首 + C尾 | 实际上 -> pmnm |
| 三字词 | 1 + 2 + 1 | A首 + B首+B尾 + C尾 | 输入法 -> johy |
| 三字词 | 1 + 1 + 2 | A首 + B首 + C首+C尾 | 这种人 -> yhoo |
| 四字词+ | 1 + 1 + 1 + 1 | A首 + B首 + C首 + Z首 | 社会保障 -> ioof |

原则：遵循仓颉五代顺位，动态消除同字内冗余码，首字不超过 2 码。
